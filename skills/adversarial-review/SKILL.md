# Adversarial Review

外部评审缺位（无 codex / 没人帮看）时，对自己刚写的 slice 跑多镜头对抗审，再正确处置 findings。自审有系统性盲区——不走这一步，一个假覆盖测试会绿着混过、一个真 bug 被「这是设计意图」打发、一个限流失败的镜头被当成「0 finding=干净」。

## 触发场景

- 写完一个 slice 正要 push，没有外部评审。
- 高风险改动：删除型 / 搬迁型 / 改了回放或公式不变量。
- 一轮 review 出了 N 个 finding，要决定改哪些。

## 流程

1. **派 2-3 个镜头**（本仓限流敏感，并发 ≤2-3），每个镜头独立视角：
   - **确定性 / 回放**：同输入是否同输出？RNG 消费顺序 / 计数有没有被悄悄改？公式有没有引入浮点？（CCZ 第一镜头，永远要有）
   - **契约 / 校验**：内容包 validator 是否 fail-closed？schema / 版本轴是否一致？command/event 契约字段是否漂移？
   - **测试效力 / 集成**：测试是真断言行为，还是 marker 盖一下就绿？删/改的测试是不是某存活代码的唯一覆盖？
   - （涉及文案 / UI 时加一个 copy 镜头）
2. **每条 finding 标 severity**：P1=数据/确定性正确性 → **合前必修**；P2=该修；P3=锦上添花。
3. **处置纪律**：
   - 数据 / 确定性正确性 finding **合并前修**，不准标「gap 延后」；只有闭环缺口可 flag-OFF 延后并钉死四处。
   - 不把一条 finding 当「设计意图」打发——要么拿证据证明它不成立，要么修。
   - 一个镜头返回 null / 报错 / 被限流 ≠「clean」；重跑或换串行，别把沉默当通过。
   - 子代理报「已删 / 不存在 / base 不对」之类阻断结论前，逐站点 grep/read 核实，避免假 blocker 唬停你（子代理可能在 stale worktree 或错 cwd）。
4. **不把 review 噪音倒进 PR**；只把确认的 finding 转成改动或显式 backlog。

## CCZ 暗雷

- 同进程回放确定性绿 ≠ 跨版本回放安全：roll 顺序/计数变了 ReplayContractTest 照样绿——确定性镜头要专门盯顺序/计数。
- `ContentValidator` 不读 events：「unknown op 被拒」是文档承诺不是现实，别误判已覆盖。
